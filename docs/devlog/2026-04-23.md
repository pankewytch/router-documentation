## Issues
1. Validation of the `named.conf` file failed.
2. Validation of reverse zone file failed.
3. While deploying Bind9 a few times to resolve these validation issues, the update for apt was not resolving.

## Resolutions
1. Looking through the system logs on the router, the validation failed due to the lack of a ; on the include statement. I found that there was a missing ; after the options block but before the `includes` at the end of the config. I added this ; in and the validation for this file passed.
2. The error was found in the logs stating that the validation had failed for the reverse zone file, but no other useful information printed out. Upon consulting the documentation, I had found that name of the file had to be in terms of octet hierarchy, not how one would read an IP address. Meaning, if an subnet were 11.22.33.0/24, then the reverse zone file is looking for 33.22.11 for the order of octets. Once this was changed, the validation was successful and Bind9 successfully launched.
3. The system-resolved was updated to point to 127.0.0.1 before Bind9 could launch due to the validation issues. This issue was investigated and a dig command showed there was a failure in the DNS resolution and that failure occurred when looking for a server on 127.0.0.1. This issue resolved itself when when Bind9 was successfully configured. In addition, the script was modified so the system-resolved tasks would only run AFTER the Bind9 successfully launched so this will no longer be an issue in the future.

## Decisions Made
- Use DDNS (kea-dhcp-ddns) to update Bind9 and manage internal device names. Also reduces dependency on static IP addresses
- Upon research, Postgresql to manage client names and static reservations to simplify the config. The goal is to name every IoT device as they are migrated from the old to the new network.
- Postgres will run on a docker container with a stateful anchor on disk for the data
- Postgres will be available to other applications, a logical db and user will be created for kea
