## Issues
1. Mongo db version 4.4 would not launch on the raspberry pi. It failed with a code 132.
2. Unifi application would not launch the tomcat server. Error logs detailed there was a lack of appropriate database permissions.
3. Ubiquiti equipment was stuck in `adopting` and would not adopt.
4. Nftables config push broke ssh access from the pi runner and to my local dev machine, effectively according to the logs via the COM1 port, all port 22 connections to the router were dropped.
5. When deploying the kea config, there was issues with validation and I was not sure what those issues were.
6. Leases were not being issued to network clients.

## Resolutions
1. Mongo DB version 4.4 would not launch. The error clearly stated that the Pi's CPU was not supported. I went to the documentation and found that the last version of Mongo that supported this DB was 4.2. I tried reloading the docker image and it still failed. Upon further reading, the patch version 18 (4.2.18) was the last version of Mongo DB to support the Pi's CPU. This was modified in the docker compose yml file. The database then loaded successfully and login attempts via `docker exec` were successful.
2. In order to resolve database permission errors for unifi_stat and unifi_audit databases, the readWriteAnyDatabase role was granted to the unifi user. This is broader than ideal but is acceptable since MongoDB is not exposed outside the Docker network and only the Unifi container can reach it. This should be revisited when automating the Eris pipeline to scope permissions more tightly.to all it needed.
3. The config for unifi had to be modified to uncomment the system ip address. Then, the docker container was restarted. From there, all the devices slowly came on line after completing a few firmware updates.
4. First and foremost, I connected to the COM1 port to regain access. I then read through the nftable logs to determine that all port 22 traffic was being dropped on all connections not coming in from the LAN trunk interface. I updated the rules to allow it from the WAN port for now as this is only connected to my local network and not directly to the ISP just yet. Once I made this change and restarted the nftables service, I could once again push a tag and the pipeline sucessfully deployed the changes.
5. The script that deploys the kea configs was modified to print the full validation output into the routers journal IFF it failed validation. This led me to realize 2 issues.
    - Validating the config in the repo directory was failing because the `_kea` system user did not have access to the file in that directory. The script was changed to create a temp config in the `/var/lib/kea/` directory and then delete it on both success or failure.
    - The memfile implemented was from older documentation. When Kea is installed, it defaults to using a csv file for IPv4 and IPv6 leases. I changed the memfile to point to the IPv4 csv file and this then allowed kea to restart successfully.
6. Network clients were not receiving leases. The first area to check was the journal, specifically nftable's logs, as this was a good starting point given the other issues iwth it today. It was found that there was a significant amount of packets dropped from network clients on UDP port 67. From there, the nftables config was investigated and there was a section missing allowing for sport on that port, not just dport. Once that was pushed to the repo, a tag was pushed, and the git runner completed a deployment, the clients were immediately issued leases from Kea.
