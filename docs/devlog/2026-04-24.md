## Issues
1. When creating the password files, the permissions did not allow for postgres to read them when initializing the docker container for creating the kea user.
2. When the migration via the built in kea utility was run, there was an error as it could not find `psql` in its path. It failed as a result.
3. Reservation script failed due to accessing a field that was not a the top level.
4. When processing the reservations individually, `jq` failed to parse certain fields and failed.
5. Reservation deploy script failed on a cryptic `sed` error.
6. The `on conflict` portion of the upsert produced a sql error complaining about a lack of uniqueness.
7. Certain errors, like those from `sed` and `jq` were only visible on the Github actions and not in the logs on the router.
8. The DDNS update was failing due to an existing record in the zone file for Bind9.
9. The journal file could not be created due to lack of permissions in `/etc/bind/zones` according to the `named` logs.

## Resolutions
1. Permissions were never explicitly defined when creating the password text file. It was assumed the permissions would be sufficient for the kea user to be created with the stored password. Upon seeing this was not the case in the postgres logs from the docker container, permissions for this file were set to 644 and that resolved the issue.
2. This was a simple fix, since `psql` was required in its path for the migration to run, `postgresql-client`. This solution was immediately obvious, although admittedly overlooked before deploying, because Postgres was not installed locally and instead, on a docker container, so there was no access to any postgres functionality on the path locally on the router. It was briefly considered to run this in the docker container but then, the opposite would have been true, no access to the kea admin, and that would have been a larger problem to solve. Additionally, this will be needed for any major updates to kea to run the migration again.
3. When the deploy reservation script first ran, there was an error from `jq` in the iterator because the top level object in the JSON is a list named `"reservations"`. I did not notice when writing that I was iterating through `.[]` and not `.reservations`. The loop worked after this change was implemented.
4. It was quickly determined through the logs that the field `jq` failed to parse was `hw-address` on the first reservation in the JSON. The error indicated an invalid operation so it was assumed it was interpreting the `-` char as subtraction instead of as part of the string. The field was double quoted when accessed and `jq` could parse them. A brief note, the `-` was used to match the field names used in the kea-dhcp config file so the docs could still be consulted about specific fields.
5. This was a tough problem to solve. The error stated that there was a delimiter issue, specifically when parsing the password for the kea postgres user. Upon looking at the password itself and some documentation for openssl to determine allowable chars, the password contained `/` so when doing `s/thing1/thing2/g`, it was breaking. To fix this, the `|` was used instead. To be frank, I was unaware the `|` char could be used in place of `/`. After the switch was made to `s|thing1|thing2|g`, the `sed` completed successfully.
6. At first, the mac address and the identifier type were use to determine uniqueness but these were clearly not guaranteed to be unique. So, the table `hosts` was investigated, specifically the dll. It was discovered that these were indeed not a unique tuple, but, there was a uniqueness constraint and index on those 2 fields AND subnet id. Once subnet id was added to the on conflict and removed from the subsequent update, the script inserted record.
7. A line was added at the top of the deploy script. The stdout was piped into the logs with the tag deploy. It is not the cleanest solution and something better should be implemented in the future, but this unified the logs so I could rely on the router logs alone to catch any errors during deployment.
8. The issue was clear here. The record still existed in the reverse zone file and had to be removed. It is a strict binary, either the DDNS handles the IP mappings or the reverse zones do. The only obvious exception is the interface IP as that is NEVER assigned via DHCP.
9. This was a difficult issue to solve. First, it was thought tha that the file was not owned by the bind user, so a `chown` for the `/etc/bind/zones` was added into the deploy-dns-internal script. The issue persisted. Next, the permissions were explicitly defined for the `/etc/bind/zones/` directory. They were defined as 775, this was enforced through the deployment script via `chmod`, and another deployment was completed. The issue persisted. At this point, there was only one culprit left, AppArmour. This could be the only reason that the permissions were stopping the creation of a journal file because 775 would have been more than sufficient to allow write to the bind user. The AppArmour rules where cat'ted for the bind user and it was found bind only has write permissions to the `etc/bind/*` directory, but no read. At this point, there was a fork in the road, add maintaining the ruleset of AppArmour to the deployment pipeline, or use a directory for the zones that AppArmour allowed bind to write to, specifically `/var/lib/bind/`. The latter was chosen to simplify the deploy process in addition to lowering the overhead of maintaining the pipeline in the future. After the zone files were moved to `/var/lib/bind/zones/`, the journal files were created. A client on the network had their ethernet interface reset in addition to wiping the lease from kea-dhcp. It was confirmed that the lease was reissued, there was a FQDN, present in the logs for the DDNS, and the journal file was created and written to in the named logs.

## Decisions Made
- Postgres to use named volume in docker as all reservations can be quickly redeployed and source of truth is the config json, not Postgres
- Start with a plaintext password file while the router is still behind the current network - low risk
- Discovered AppArmor was restricting Bind9 write access to `/etc/bind/zones/`,  resolved by moving zone files to `/var/lib/bind/`zones/ which AppArmor permits Bind9 to write to
